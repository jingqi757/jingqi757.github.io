题目说orw，先了解什么是orw

在ctf的pwn题中一般有两种函数调用方式实现沙盒机制，第一种是采用prctl函数调用，第二种是使用seccomp库函数

prctl

seccomp

但是这道题什么限制都没开，但是用不了system的系统调用，只能用0，1，2，60。一开始想的是直接shellcode在缓冲区，发现给的栈溢出空间太小，只能跳到mmap里面了

先理一下大体思路：在栈空间上写read到mmap并且jump的代码，然后在这个函数的ret上jump到栈上执行，这里要注意怎么jump到栈上，我之前有误区，以为rsp向上，栈空间会被清除，现在才知道只是rsp变了，栈上数据不会被清除

因为栈空间不够，所以要先写到mmap里面再jump到mmap，一开始进行read的操作，read完就要跳mmap，然后ljsut填满栈空间到ret，执行完ret弹出地址到rip，这个时候rsp上移了8位，jmp_rsp是到返回地址上面8位的，所以要有asm(‘sub rsp,0x30; jmp rsp’)才能跳到栈开头，然后就是等待orw输入到mmap的空间，紧接着执行orw。

jmp rsp的技术是栈枢轴，不需要知道具体的地址

  1. 返回地址被覆盖为 jmp rsp 指令的地址。
  2. 当函数执行 ret 时，程序跳到 jmp rsp。
  3. 此时 rsp 刚好指向返回地址之后的位置（也就是你放 Shellcode 的地方）。
  4. 程序顺着执行。

但是这里read的不够，所以跳到了mmap，下面是gemini给我解释的真正的栈枢轴

真正的“栈枢轴”（Stack Pivoting）通常配合 ROP (Return Oriented Programming) 使用：

  1. 真正的 Stack Pivoting：利用 pop rsp; ret 或 leave; ret。
  2. 作用：把栈指针 rsp 强行改到一个你提前准备好的可读写内存区（比如 BSS 段或 mmap 出来的空间）。
  3. 结果：在那块空间里，你执行的是 ROP 链（一连串的 pop rdi; ret 等），而不是 Shellcode。ROP 链是程序原本的代码段，所以 NX 无法拦截它。

SROP

你在做 ctfshow 或其他比赛时，如果看到以下特征，就要考虑 SROP：

1.
2. 寄存器传参 Gadget 严重缺失（比如没有 pop rsi 或 pop rdx）。
3. 溢出空间非常充裕（几百个字节）。
4. 静态编译的程序（容易找到 syscall）。

总结

SROP 就是利用了操作系统的“健忘”和“死板”。它把复杂的 ROP 链简化成了一次大规模的“寄存器整容”。

程序里是 scanf(“%llx %d”)。它需要看到：地址 + 空格 + 数字。你的写法 f’{hex(addr)}{data}’ 会把它们连在一起。例子：如果地址是 0x400767，数据是 182。你发送的是：”0x400767182”scanf 会把这整串都当成十六进制地址读走。由于 scanf 没读到第二个数字（%d），它会一直卡在那里等着，或者直接出错返回。

还有就是当你使用 f’{text_addr}’ 构建发送给程序的字符串时，Python 默认会把它转成十进制字符串。，所以要u32（.ljust）

pwn89

花式栈溢出

首先看汇编代码

这里面有canary值需要绕过，正常来说需要爆破或者什么的，但是这道题是创建了一个子线程，fs段在子线程中距离栈非常近，可以直接把fs段也用aaaa溢出了，就可以绕过canary

在溢出的过程中还可以泄露出libc地址，并且把栈迁移到一个干净的地方，比如bss段，利用 readelf -S ./pwn89

但是要再向后偏移几百字节，因为bss段前面有一些全局变量

然后就可以read到bss段，在p64(bss_addr-8)和p64(leave_ret)中间就是我们利用的rop，接着ljust填充0x2000溢出到fs段，这样在函数结束前校验的时候都是aaaa，然后我们再send到bss段完成攻击

这里有几个坑需要注意，一个是sleep(0.5),这里是为了防止两个send内容在粘在一个里面，以后可以带上这个习惯，还有就是用的是gadget，libc6_2.27-3ubuntu1.6_amd64的one_gadget偏移（不可以libc.dump(‘one_gadget’)），在这个版本的 Libc 中，常用的 one_gadget 偏移有三个：0x4f2a5, 0x4f302, 0x10a2fc，可以直接用，但是我用system构造就不行，明明也选对了libc版本，为什么呢。

pwn90

Canary 绕过 结合 PIE（地址随机化）局部溢出

先看ida

QWORD是八字节，所以buf5再buf向上40，注意是40，第一次写的时候习惯写成了0x40，结果一直报错…

因为canary第一个字节是\x00，为了正常输出，所以我们要把第一个溢出为正常字符。然后再利用canary=data[41:48]
canary=b’\x00’+canary补完全，这里已经是八字节的情况下为什么还要canary=u64(canary.ljust(8,b’\x00’))呢，ai是这样说的

即使因为 Canary 内部提前出现了 \x00 导致你只抓到了 3 字节，它也会自动补齐到 8 字节。虽然这样拿到的 Canary 是错的（会导致后面爆破失败），但至少脚本不会崩溃，你可以通过循环多次尝试来完成泄露。

接着就是PIE，可以看到有后门函数但是开了PIE，意味着text代码段还是也是随机的，不可以直接跳到后门函数，但是PIE不会改变后三位的数字，所以我们可以利用局部覆盖，这里面只用了两个十六进制位是因为main原本的ret返回地址和后门函数很近，所以第三个大概率一样。思考了一下，那我覆盖三位呢，并不行，因为内存的最小单位是字节，所以必须两位两位写，这里如果要覆盖三位就要带上第四位，但是第四位是随机的

为什么不能是3e呢？我一开始写的3e，报错了，后来想了想可能还是因为64位的栈平衡，3e是push ebp，栈不平衡了。
